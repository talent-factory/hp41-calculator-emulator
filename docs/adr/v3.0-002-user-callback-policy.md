# ADR-002: User-Callback Re-entrancy Policy — Strict-Reject Nested INTG/SOLVE/DIFEQ

## Context

HP-41C Math Pac I (HP 00041-90034, 1979) implements three solver programs — `INTG`
(numerical integration), `SOLVE` (root finding), and `DIFEQ` (ODE solving) — that
each accept a user-defined program label (e.g., `LBL "MYFN"`) as a callback function.
The solver calls the user function repeatedly, once per sample point in its iteration
loop. For INTG, up to 32768 evaluations may occur per call (ADR-004).

**The re-entrancy problem:**

The user callback function is a regular HP-41 program that can execute any HP-41 op,
including `XEQ "INTG"`, `XEQ "SOLVE"`, or `XEQ "DIFEQ"`. When the user callback
itself invokes one of the solvers, a nested call occurs:

```
op_integ (outer) → run_loop → user LBL "MYFN" → op_integ (inner) → ???
```

On real HP-41C hardware with Math Pac I, the nested call behavior is undefined. The
Math Pac I programs (`INTG`, `SOLVE`, `DIFEQ`) use registers R00–R07 as scratch storage
during iteration (see D-30-01 / D-30-06 in `docs/hp41-math1-divergences.md`). When the
inner INTG call starts, it overwrites R00–R07 with its own scratch values, corrupting
the outer INTG's iteration state. The outer solver then reads its corrupted state on
the next iteration and produces a wrong numerical result. In extreme cases, the
calculator may hang or require a power cycle.

**The OM's position:**

HP 00041-90034 (1979) warns against nested calls but does NOT define an error behavior
for the nested case. The OM language is: "Do not use INTG within a function called by
INTG" (paraphrase — exact page reference to be verified against physical copy). The OM
provides the prohibition but leaves the consequences as "undefined behavior" on real
hardware.

**The emulator's `callee_state` model:**

`hp41-core`'s `CalcState` carries `integ_state: Option<IntegState>` and
`solve_state: Option<SolveState>` fields (both with `#[serde(skip)]` — transient, not
persisted). These fields hold the outer solver's mid-iteration scratch state. During a
normal (non-nested) INTG call: `integ_state` is `None` at op entry, set to `Some(...)`
during the call, and cleared to `None` on completion or error. A nested INTG call would
find `integ_state` is already `Some(...)` at op entry.

**The Free42 oracle situation (Pitfall 19):**

Free42 (Thomas Okken, GPL v2+, `https://github.com/thomasokken/free42`) implements
its own `do_intg` / `do_solve` / `do_difeq` functions in
`free42/common/core_math1.cc`. Free42's re-entrancy handling was inspected as a
sanity-check oracle during Plan 28-01 research to understand the behavior space.

This sentence is the per-file header policy for all `hp41-core/src/ops/math1/*.rs`
files. The CI enforcement script (`scripts/check-free42-contamination.sh`) lands in

## Decision

**The strict-reject policy is locked: nested INTG/SOLVE/DIFEQ returns `HpError::InvalidOp`.**

The decision was made in Plan 28-01 Task 1 (locked 2026-05-16). The policy in detail:

At the entry of `op_integ`, `op_solve`, and `op_difeq`, the implementation checks
whether a nesting condition exists by examining `CalcState`:

```rust
// Example for op_integ (pseudocode):
if state.integ_state.is_some() || state.solve_state.is_some() {
    return Err(HpError::InvalidOp("NESTED INTG/SOLVE NOT ALLOWED".into()));
}
```

This check fires BEFORE any state mutation. If nesting is detected, the op returns the
error immediately with the message `"NESTED INTG/SOLVE NOT ALLOWED"` and the outer
solver's `integ_state` / `solve_state` is not modified. The user-program callback (the
inner INTG/SOLVE/DIFEQ that triggered the nesting) sees the `HpError::InvalidOp` and
propagates it up through the `run_loop` callback mechanism to the outer solver, which
can then handle it as an evaluation error (typically: abort the outer integration and
surface the error to the user).

The error message `"NESTED INTG/SOLVE NOT ALLOWED"` is the locked text. This exact
string appears in `hp41-core/tests/math1_user_callback.rs` as a test assertion. Any
future change to this string is a breaking change requiring a test update.

**The strict-reject is intentionally stricter than real hardware:**

Real HP-41C hardware with Math Pac I does NOT error on nested calls — it silently
produces wrong answers (or hangs). Our strict-reject is more user-friendly: users who
accidentally write nested-INTG get an immediate, clear error message rather than a
mysterious wrong numerical result. The trade-off (OM-compatibility vs. user safety) is
explicitly accepted, documented in this ADR and in the divergence catalog entry D-30-06.

**Free42 disclaim (Pitfall 19 — second appearance in Footnotes section [^4]):**

Algorithm independently re-derived from HP Math Pac I Owner's Manual 00041-90034 (1979); Free42 source consulted only as sanity-check oracle, not copied.

The re-entrancy detection logic (checking `integ_state.is_some()`) is structurally
obvious from the `CalcState` model — it follows directly from the decision to store
mid-iteration state on `CalcState` rather than on the stack frame. Free42's
implementation was inspected to understand the behavior space, not to copy the
detection mechanism.

## Consequences

### Positive

- **Deterministic error behavior:** a user who accidentally writes nested-INTG sees
  `HpError::InvalidOp("NESTED INTG/SOLVE NOT ALLOWED")` immediately, on the first
  nested call. No wrong-number debugging required.
- **Bisectable failure mode:** the `InvalidOp` path is explicit and carries a diagnostic
  message. Test assertions on the exact message text make regression testing trivial.
- **No state corruption:** the outer solver's `integ_state` / `solve_state` is
  unmodified by the rejected inner call. The outer solver can continue (if the
  outer caller chooses to handle the error gracefully) or abort cleanly.
- **Positive regression test:** `hp41-core/tests/math1_user_callback.rs` includes an
  explicit test that verifies the nesting-rejection behavior — the test asserts that
  `op_integ` returns `Err(HpError::InvalidOp(...))` when called from inside a callback
  that is itself executing inside `op_integ`. This makes the strict-reject policy
  testable and version-controlled.
- **Defense-in-depth:** the `call_stack` 4-deep cap (from `Op::XeqInd` precedent in
  v2.2 at `hp41-core/src/ops/program.rs:479`) catches pathological nesting BEFORE
  the strict-reject check fires in deeply-nested scenarios. Both gates together provide
  defense-in-depth against the nested-solver failure mode.

### Negative

- **Stricter than OM:** the policy rejects a case the OM only warns about. A user who
  writes nested-INTG intentionally (for some theoretical reason) will hit a hard error
  rather than wrong-but-plausible numbers. This is documented as a behavioral policy in
  the divergence catalog (D-30-06 in `docs/hp41-math1-divergences.md`) and flagged as
  "emulator stricter than OM" — the accepted trade-off.
- **Not hardware-faithful for the nested case:** on real HP-41C hardware, nested INTG
  does NOT error — it silently corrupts and produces wrong answers. Testers who run
  the emulator against real hardware specifically to test nested-INTG behavior will
  observe a difference. This is the one known case where the emulator is intentionally
  not hardware-faithful (the OM's prohibition makes the hardware behavior "wrong by
  design"; we enforce the prohibition rather than replicating the wrongness).

### Neutral

- The `state.call_stack` depth cap and the `integ_state.is_some()` check are two
  independent guard layers. Neither replaces the other. The `call_stack` cap catches
  deeply-nested *program* re-entry; the `integ_state` check catches *solver* re-entry.
  Both can fire on the same pathological input; the order is: `call_stack` cap first
  (checked in `op_xeq`), then `integ_state` check (at `op_integ` entry).
- The Free42 contamination guard (`scripts/check-free42-contamination.sh`, Phase 32
  / QUAL-05) will grep each `hp41-core/src/ops/math1/*.rs` file for the per-file
  header sentence. This ADR's Decision section contains the sentence; the Phase 32
  audit will also check the source files.

## Alternatives Considered

### Option B: Silent-Tolerate (Match OM "Undefined Behavior" on Nested Call)

The rejected alternative is described verbatim in 28-CONTEXT.md C-28.2 (locked 2026-05-16):

> **C-28.2 (ADR-002):** User-callback re-entrancy — strict-reject nested INTG/SOLVE/DIFEQ
> at op entry with `HpError::InvalidOp`.
>   - **Why:** matches Math Pac I OM 1979 Hardware-Verhalten; simplest invariant; avoids
>     4-deep `call_stack` overflow + cleanup-on-error complexity.

**Additional context on why Option B was rejected:**

Silent wrong answers are the worst failure mode for a calculator emulator. A user spending
hours debugging "why is my integral wrong?" without realizing the cause is nested-INTG is
a UX disaster. The strict-reject surfaces the error immediately with a diagnostic message.
The OM warns against nesting; enforcing the warning at the engine level is more
user-friendly than ignoring it.

Additionally, "silent wrong answers" cannot be tested — there is no well-defined correct
answer for nested-INTG on real HP-41C hardware (the output is implementation-dependent
and may differ between hardware units, firmware revisions, and temperature conditions due
to timing). Strict-reject is testable: the `InvalidOp` result is deterministic and
verifiable.

### Option C: Snapshot/Restore `integ_state` Around the Nested Call

A third option was considered: before the inner INTG call, snapshot the outer
`integ_state` and `solve_state`; after the inner call completes, restore the outer
state. This would allow nested-INTG to work without corrupting the outer solver.

Rejected for two reasons:

1. **Structural impossibility for the singleton-field model:** `integ_state` is a single
   `Option<IntegState>` field on `CalcState`. There is no stack of `IntegState` values —
   only one can be active at a time. Making nested-INTG work would require either (a) a
   `Vec<IntegState>` stack on `CalcState` (adding complexity and memory overhead for a
   case the OM explicitly prohibits) or (b) deep-copying `CalcState` before each nested
   call (the 30 KB × 1000 samples re-clone catastrophe that C-28.5 rejected for the
   user-callback mechanism in general).

2. **OM prohibition:** the OM explicitly warns against nested calls. There is no
   obligation to make nesting work — the OM's prohibition implies the user is responsible
   for not creating this situation. Making it work (Option C) would contradict the OM's
   explicit warning and give users false confidence that nested-INTG is supported behavior.
