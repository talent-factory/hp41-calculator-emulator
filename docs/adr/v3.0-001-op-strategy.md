# ADR-001: Op-Strategy — One Op Variant per Math Pac I Function

**Status:** Locked 2026-05-16
**Owner:** Plan 28-01 Task 1
**Requirement refs:** XROM-01..09, Pitfall 1 (28-RESEARCH.md §"Decision 1")
**Downstream consumer:** `hp41-core/src/ops/mod.rs` (`Op` enum and `dispatch()` exhaustive match),
`hp41-core/src/ops/program.rs` (`execute_op()` exhaustive match),
`hp41-cli/src/prgm_display.rs` (CLI program-step display formatter),
`hp41-gui/src-tauri/src/prgm_display.rs` (GUI program-step display formatter)
**ADR write-up prose:** Phase 30 / DOC-07 (this is the narrative ADR; the lock happened in Plan 28-01)

---

## Context

HP-41C Math Pac I (HP 00041-90034, 1979) adds approximately 40 new callable functions
exposed via ~55 XEQ-by-name entry points. Examples include `INTG`, `SOLVE`, `POLY`,
`DIFEQ`, `FOUR`, `MATRIX`, `SINH`, `COSH`, `TANH`, `ASINH`, `ACOSH`, `ATANH`,
`XTOA`, `ATOX`, triangle solvers (`TRIG-SSS`, `TRIG-SAS`, ...), and complex-stack
operators (`C+`, `C-`, `C×`, `C÷`, `MAGZ`, `Z↑N`, etc.).

Before implementing these in `hp41-core`, the architecture team faced a strategic choice
about how to represent them in the `Op` enum:

**Option A (chosen):** Add one `Op` variant per Math Pac I function.
Each function gets a dedicated Rust variant: `Op::Sinh`, `Op::Cosh`, `Op::Matrix`,
`Op::Integ`, `Op::Solve`, `Op::Poly`, `Op::Difeq`, `Op::Four`, etc. The `Op` enum
grows by approximately 40 variants.

**Option B (rejected):** Add a single `Op::XromCall(u16)` variant that table-dispatches
by a packed `(module_id, function_id)` u16. All Math Pac I functions are routed through
this single arm; the dispatch table is a `&[(u16, fn(&mut CalcState) -> HpResult)]`
slice.

The critical constraint is the **4-place exhaustive-match invariant**: in the v1.0–v2.2
codebase, every `Op` variant must appear in four exhaustive match arms:

1. `dispatch()` in `hp41-core/src/ops/mod.rs` — the primary runtime dispatch
2. `execute_op()` in `hp41-core/src/ops/program.rs` — the program-execution dispatch
3. `op_display_name()` in `hp41-cli/src/prgm_display.rs` — CLI step formatter
4. `op_display_name()` in `hp41-gui/src-tauri/src/prgm_display.rs` — GUI step formatter

This invariant is enforced by the Rust compiler: a non-exhaustive match arm on `Op` is
a compile error. Every `Op` variant must be handled in all four locations before any
crate that imports `hp41-core` can compile. This has been the single most reliable
bug-catching mechanism in the codebase since Phase 1 — if a new variant is added to
`dispatch()` but forgotten in `execute_op()`, the build fails before any test runs.

Option A preserves this invariant with ~40 new explicit arms per match. Option B
collapses all Math Pac I functions into a single `XromCall(u16)` arm in each match —
one arm handles all 40+ functions via a runtime lookup table.

---

## Decision

**Op-strategy A is locked: one `Op` variant per Math Pac I function.**

The decision was made in Plan 28-01 Task 1 (locked 2026-05-16) as part of the Phase 28
XROM framework setup. The rationale in three parts:

**1. Preserves the compile-time exhaustive-match invariant.**

The 4-way exhaustive match has caught real bugs in v1.0–v2.2. Three documented examples:

- Phase 25 (Plan 25-04): `Op::Xeq(String)` was added to `dispatch()` but its
  `execute_op()` arm was initially missing; the build caught it before any test ran.
- Phase 22 (Plan 22-02): `Op::Catalog` had a missing `prgm_display.rs` arm; the GUI
  build caught it when Phase 26 synced GUI to CLI.
- Phase 24 (Plan 24-01): All 11 `*Ind` variants needed `execute_op()` arms; the
  compiler enforced each one.

Option B would collapse this to a single `XromCall(u16)` arm in all four match
locations. A missing function_id in the dispatch table becomes a runtime error
(wrong-answer or panic in the XROM table lookup) rather than a compile error. The
compile-time → runtime regression in error-detection latency is unacceptable given the
~40 new functions that must all be wired correctly.

**2. Per-Op test fan-out maps naturally to per-variant coverage.**

Per-Op test count ≥ 5 (Pitfall 16 from 28-RESEARCH.md) applied to ~40 new variants
yields ~200 new test functions. Under Option A, each test targets a specific `Op::*`
variant by name — bisecting a test failure is trivial (the failing test name includes
the Op variant). Under Option B, all 40+ functions would need to route through a single
`Op::XromCall(u16)` test fixture — a mega-test with table-driven cases where one
function's regression can be hidden by another function's pass.

**3. CATALOG and prgm_display derivation is structurally simpler.**

The upcoming Phase 31 GUI feature CATALOG-2 (listing all loaded XROM modules) derives
its listing from the static `&[XromModule]` table where each entry references its
`Op::*` variant directly. Under Option A, the derivation is a static iterator over
known variants — no runtime lookup, no indirection. Under Option B, the GUI listing
would require a parallel runtime function_id-to-display-name lookup table, duplicating
the function-id semantics in a different layer.

The CLI ↔ GUI parity invariant (D-25.6 from v2.2) requires both surfaces to resolve
identical XEQ-by-name strings to identical `Op` variants. Under Option A, parity is
trivially verifiable via grep — `grep "XEQ \"SINH\""` must appear in both
`prgm_display.rs` copies. Under Option B, both surfaces must share an identical
function_id lookup table, adding a new synchronization requirement between
`hp41-cli` and `hp41-gui` that the build does not enforce.

## Consequences

### Positive

- Compile-time exhaustive-match invariant preserved for all ~40 new Math Pac I ops.
  Missing variants cause build failures, not runtime surprises.
- Per-Op test functions (`test_sinh_positive`, `test_sinh_negative`, etc.) are
  individually bisectable — a failing test directly identifies the broken variant.
- `op_display_name()` arms in both `prgm_display.rs` copies are explicit and
  compiler-enforced — no "unknown op" fallback needed.
- CLI ↔ GUI parity verifiable via grep: `grep "Op::Sinh"` must appear in both files.
- Static `&[XromModule]` table for CATALOG-2 derivation is straightforward.

### Negative

- The `Op` enum grows by approximately 40 variants. This is a purely additive change
  (no existing variants are modified) but increases the enum's size in memory by
  approximately 40 discriminants.
- The `criterion bench/dispatch_overhead.rs` floor is `< 200 ns/op` (v2.2 baseline:
  ~65 ns/op). Phase 32 / QUAL-08 must verify the floor still holds after ~40 new
  variants are added. Typical headroom (65 ns baseline vs. 200 ns floor) is comfortable,
  but the bench must confirm this after Phase 28 ships all variants.
- Each new Op variant requires four edits across four files (two in `hp41-core`, one in
  `hp41-cli`, one in `hp41-gui`). Phase 28 owns the first two; Phase 29 and Phase 31
  own the remaining two. Tracking which variants are "partially landed" during the
  Phase 28→29→31 pipeline requires discipline. The `prgm_display.rs` compile error
  is the enforcement mechanism.

### Neutral

- The XROM resolver chain (`xrom_resolve` fires LAST per C-28.4 / Pitfall 1) does not
  care which Op-strategy is chosen — it returns an `Op` either way. The resolver chain
  extension is orthogonal to the enum-shape decision.
- Option A and Option B produce identical runtime behavior for correctly-implemented
  functions. The difference is entirely in error-detection latency and structural
  maintainability, not in numerical output.
- `hp41-core`'s `#![deny(clippy::unwrap_used)]` constraint applies identically under
  both strategies; no new unsafe patterns are introduced by Option A.

## Alternatives Considered

### Option B: `Op::XromCall(u16)` Table Dispatch

The rejected alternative is described verbatim in 28-CONTEXT.md C-28.1 (locked 2026-05-16):

> **C-28.1 (ADR-001):** Op-strategy A — one `Op` variant per Math Pac I function.
> Rejected Option B (`Op::XromCall(u16)` table dispatch) preserves the 4-way
> exhaustive-match invariant that has caught dozens of bugs since Phase 1.
>   - **Why:** zero new test surface, compile-time `prgm_display.rs` coverage check,
>     structurally identical to v1.1's `synthetic_byte_to_op` resolver.

**Additional context on why Option B was rejected:**

Option B's apparent advantage — fewer match arms — is a false economy. It replaces
a compile-time guarantee with a runtime guarantee. The 4-way exhaustive match has been
the most reliable single quality gate in the v1.0–v2.2 codebase; collapsing it to a
single arm loses the catch. The additional notes:

- **Branch-prediction noise:** a runtime function_id dispatch table adds an indirect
  branch (table lookup by u16 key) where Option A has a direct match arm. At ~65 ns/op
  v2.2 baseline, the indirect branch represents approximately one cache-miss budget.
  Option B may push the dispatch overhead toward or past the 200 ns/op floor without
  the benefit of explicit per-Op benchmarks per variant.

- **`synthetic_byte_to_op` precedent:** v1.1's `synthetic_byte_to_op` (Phase 12) uses
  exactly Option A for the 24-entry safe subset of synthetic programming — one explicit
  match arm per byte value. The mechanism works, is tested, and is structurally
  identical to what Option A proposes for Math Pac I. There is no precedent in the
  codebase for Option B's table-dispatch pattern.

- **XROM shadowing test (`tests/xrom_shadowing.rs`):** under Option A, the shadowing
  test iterates `MATH_1.ops` and asserts `key_to_op(name)` / `builtin_card_op(name)`
  return `None` first — the Op variants are explicit and can be enumerated. Under
  Option B, enumerating function_ids to check for shadowing requires walking the
  dispatch table, which is harder to grep-audit.
