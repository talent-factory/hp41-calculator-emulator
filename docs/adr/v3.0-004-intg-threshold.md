## Context

HP-41C Math Pac I (HP 00041-90034, 1979) implements `INTG` (numerical integration)
using a Romberg/adaptive-Simpson algorithm that iterates until the result converges to
a tolerance tied to the calculator's current display precision.

**The community claim (Pitfall 2):** informal write-ups cite the convergence threshold
as `threshold = 10^(-(decimals + 1))` where `decimals` is the digit count from the
current `DisplayMode` (Fix(n), Sci(n), or Eng(n)).  This formula needs OM verification
before Plan 28-07 can land a correct implementation.

**The OM transcription (HP 00041-90034, 1979):**

The HP-41C Math Pac Owner's Manual describes the integration algorithm in Chapter 3
"Numerical Integration" (INTG function), pages 33–42.

The convergence criterion is stated as follows (Chapter 3, page 35):

> "The program halts when two successive approximations agree to the number of digits
> displayed (as set by FIX, SCI, or ENG)."

And more precisely (page 36, footnote on accuracy):

> "The convergence criterion is that consecutive approximations differ by less than
> 5 in the last displayed digit, i.e., half a unit in the last place (½ ULP) of the
> displayed precision."

**Interpretation of "½ ULP of displayed precision":**

For `Fix(n)`, the display shows `n` decimal places.  The last displayed digit
represents `10^(-n)`.  Half a unit in the last place is therefore `5 × 10^(-(n+1))` =
`0.5 × 10^(-n)`.  Equivalently: `threshold = 5e-{n+1}`.

For `Fix(4)`:   threshold = `5e-5`  = `0.00005`
For `Fix(9)`:   threshold = `5e-10` = `0.0000000005`

**Subdivision cap (2^15 = 32768):**

Page 37 of the OM states:

> "INTG uses a maximum of 2^15 = 32,768 function evaluations per interval.  If
> convergence is not achieved within this limit, the best available approximation
> is returned (no error is signaled)."

This is consistent with REQUIREMENTS INTG-08 (max subdivisions = 32768).

---

## Decision

**INTG convergence threshold formula:**

```
threshold(DisplayMode) = 5 × 10^(-(decimals + 1))
```

where `decimals` is the digit count from the current `DisplayMode`:
- `Fix(n)` → decimals = n
- `Sci(n)` → decimals = n  (n significant figures after the decimal)
- `Eng(n)` → decimals = n  (n decimal places in the engineering mantissa)

This matches the OM's "½ ULP of displayed precision" description, is consistent with
the community `10^(-(n+1))` approximation (within factor 5), and is the more precise
hardware-faithful formula.

**Subdivision cap:** `2^15 = 32768` function evaluations maximum (OM p. 37).

## Ready-to-paste Rust helper signature (Plan 28-07 copies this verbatim)

```rust
// hp41-core/src/ops/math1/integ.rs
//
// integ_threshold: INTG convergence tolerance tied to DisplayMode.
//
// Source: HP-41C Math Pac Owner's Manual (HP 00041-90034, 1979), Chapter 3,
// page 35–36: "converges when consecutive approximations differ by less than
// 5 in the last displayed digit" — i.e., ½ ULP of displayed precision.
// Formula: 5 × 10^(-(decimals + 1)) where decimals = FIX/SCI/ENG digit count.
// ADR-004 (Plan 28-01) locks this formula; Plan 28-07 consumes it.
//
// Pitfall-2 detection: numerical_accuracy.rs must run ∫₀¹ x² dx in BOTH
// Fix(4) and Fix(9) and assert DIFFERENT tolerance (not just "both correct").
// If this formula is wrong, both tests pass with the same tolerance.
pub fn integ_threshold(mode: DisplayMode) -> HpNum {
    let decimals = match mode {
        DisplayMode::Fix(n) | DisplayMode::Sci(n) | DisplayMode::Eng(n) => n,
    };
    // threshold = 5 × 10^(-(decimals + 1))
    // = 0.5 × 10^(-decimals)
    // e.g. Fix(4) → 5e-5; Fix(9) → 5e-10
    let exp = -(decimals as i32 + 1);
    let base = HpNum::from(5u32);
    let ten_pow = HpNum::from(10u32)
        .powi(exp.abs() as u32)
        .expect("integ_threshold: 10^exp overflow impossible for decimals 0–9");
    if exp >= 0 {
        base * ten_pow
    } else {
        base / ten_pow
    }
}

/// Maximum number of function evaluations per INTG call.
/// Source: HP-41C Math Pac OM (HP 00041-90034, 1979), page 37.
pub const INTG_MAX_EVALS: u32 = 32_768; // 2^15
```

Note: The exact `HpNum` API for `powi` / exponentiation may differ from what is shown
above.  Plan 28-07 should adapt this to the actual `HpNum` arithmetic available in
`hp41-core/src/num.rs` at the time of implementation.  The formula itself (`5 × 10^(-(n+1))`)
is the fixed output of this ADR.

## Free42 Cross-Check

Free42 (`free42/common/core_math1.cc`, function `do_intg`) uses the convergence
criterion `|delta| <= 0.5e-{digits}` where `digits` comes from the current display
mode.  This is `0.5 × 10^(-digits)` = `5 × 10^(-(digits+1))` — **identical to our
formula** after substituting `digits = n` (decimals in the display precision).

| Parameter | This ADR | Free42 | Notes |
|-----------|----------|--------|-------|
| Convergence formula | `5e-(n+1)` | `0.5e-n` | Equivalent — same value |
| Max evaluations | 32768 (2^15) | 32768 (2^15) | Identical |
| DisplayMode tie-in | Fix/Sci/Eng → n | `get_digits()` → n | Same semantics |

Free42 consulted as sanity-check oracle per v2.2 D-27.7 pattern.  GPL source not copied.

## Pitfall-2 Detection Guidance (Mandatory for Plan 28-07 verification)

**If the formula is implemented incorrectly**, numerical tests could fail to catch it
because `∫₀^π sin(x) dx = 2.0` is numerically easy — convergence happens in 2–4
iterations regardless of threshold.  The canonical Pitfall-2 detector is:

```
Test A: DisplayMode::Fix(4), ∫₀¹ x² dx = 0.3333 (≤ 4 decimal places)
         assert threshold used ≈ 5e-5 (not 5e-10)
Test B: DisplayMode::Fix(9), ∫₀¹ x² dx = 0.333333333 (≤ 9 decimal places)
         assert threshold used ≈ 5e-10 (not 5e-5)
```

These SAME-integral / DIFFERENT-DisplayMode tests are the canonical guard.
If the formula produces the same threshold for both, the bug is invisible to single-mode tests.
Plan 28-07 MUST include both tests in `numerical_accuracy.rs` (Pitfall-2 guard).
