# ADR-005: JSON-Pipeline Shape — Separate `hp41-math1-functions.json` Sibling File

**Status:** Locked 2026-05-16
**Owner:** Plan 28-01 Task 1
**Requirement refs:** DOC-01, Pitfall 13 (28-RESEARCH.md §"Decision 5")
**Downstream consumer:** `docs/hp41-math1-functions.json` (Phase 29 D-29.1 — authored in
Phase 29, consumed read-only in Phase 30),
`scripts/docs-matrix/src/main.rs` (Phase 30 Plan 30-01 surgical extension — second
invocation renders `hp41-math1-function-matrix.md`),
`hp41-cli/src/help_data.rs` (Phase 29 — second `OnceLock` for Math Pac I help entries),
`hp41-gui/src/help_data.ts` (Phase 31 — Vite parallel JSON-import for GUI help overlay)
**ADR write-up prose:** Phase 30 / DOC-07 (this is the narrative ADR; the lock happened in Plan 28-01)

---

## Context

HP-41C Math Pac I (HP 00041-90034, 1979) adds approximately 55 XEQ-by-name entry points
to the HP-41C function vocabulary. These need a machine-readable source-of-truth — a JSON
file — that drives the `?`-overlay help listing in both `hp41-cli` and `hp41-gui`, the
function-matrix documentation, and the Op-enum ↔ JSON parity test.

The v2.2 baseline established the JSON-canonical pipeline pattern (D-25.16 / D-25.17 /
D-25.18): `docs/hp41cv-functions.json` (130 entries) is the single source of truth,
loaded via `include_str!` + `std::sync::OnceLock` in `hp41-cli/src/help_data.rs`. A
malformed JSON panics at build time (`.expect("hp41cv-functions.json is malformed")` —
D-25.17 hard-build-blocker by design). The `scripts/docs-matrix/` standalone binary
regenerates `docs/hp41cv-function-matrix.md` from the JSON; `just docs-matrix-check`
is the CI drift-catch (D-25.18 / Pitfall 8 mitigation).

Before implementing Math Pac I, the architecture team faced a strategic choice about
where to store the ~55 new Math Pac I function entries:

**Option A (chosen):** Create a separate `docs/hp41-math1-functions.json` sibling file
with the same schema as `hp41cv-functions.json`, plus one additional field per entry:
`xrom: { module: string, module_id: u8, function_id: u8 }`. The two JSON files remain
independent; `hp41cv-functions.json` is not modified.

**Option B (rejected):** Extend the existing `docs/hp41cv-functions.json` to ~185
entries (130 HP-41CV + ~55 Math Pac I) by adding the optional `xrom` field to the
schema. A single unified JSON file contains all functions; the `xrom` field is
`null` for built-in HP-41CV functions and populated for Math Pac I functions.

The critical v2.2 invariant is the **parity test** in `hp41-cli/tests/function_matrix_parity.rs`:
it asserts that every entry in `hp41cv-functions.json` maps to a known `Op::*` variant
and vice versa. This test runs in CI on every push. Under Option B, the parity test
would need to be modified to handle the mixed 130+55 file; under Option A, the existing
test is unchanged and a new parallel test covers the Math Pac I file.

The additional XROM-identity constraint: Math Pac I functions are identified by a
`(module_id, function_id)` pair that the `xrom_resolve` resolver chain uses to route
XEQ-by-name calls. The hp41cv built-in functions have no such XROM identity. A per-
function `xrom` field in the schema needs to be either always-null (for hp41cv) or
always-populated (for math1). The choice of schema affects the matrix renderer, the
parity test, and the `OnceLock` help-entry loading.

---

## Decision

**Option A is locked: separate `docs/hp41-math1-functions.json` sibling file.**

The decision was made in Plan 28-01 Task 1 (locked 2026-05-16). The rationale:

**1. Zero migration churn on the 130 existing v2.2 entries.**

`docs/hp41cv-functions.json` is not modified. The 130 entries are not re-validated
against a new schema. The v2.2 `function_matrix_parity.rs` test is not changed. The
existing CI drift-catch (`just docs-matrix-check`) continues to run against
`hp41cv-function-matrix.md` without modification. The v2.2 test infrastructure for
`hp41cv-functions.json` is a proven, stable component — Option A leaves it untouched.

**2. Clean per-file test surface.**

`hp41-cli/tests/function_matrix_parity.rs` can run independent assertions per file:
one set of 4 tests for `hp41cv-functions.json` (unchanged from v2.2), one new set of
4 tests for `hp41-math1-functions.json`. Under Option B, the parity test would need to
filter by `xrom.is_some()` to separate the two test surfaces — adding test complexity
(a conditional in the test body) for no functional gain. Mixing ~185 entries in one
test also hides which surface caused a regression: a failing entry could be a broken
HP-41CV built-in or a broken Math Pac I function, and the test output would not
distinguish the two without additional context.

**3. Matrix renderer conditional XROM-column emission.**

The `scripts/docs-matrix/src/main.rs` binary conditionally renders an `XROM` column
when the input JSON's entries carry non-null `xrom` fields. Under Option A:
- Input: `hp41cv-functions.json` → `has_xrom = false` → no XROM column → output is
  `docs/hp41cv-function-matrix.md` (bit-for-bit identical to the v2.2 matrix).
- Input: `hp41-math1-functions.json` → `has_xrom = true` → XROM column rendered →
  output is `docs/hp41-math1-function-matrix.md` (new file with XROM `Math 1 / 7-N`
  column per entry, where 7 is the locked module_id and N is the function_id).

Under Option B, the matrix renderer would need to either (a) render all entries with
a nullable XROM column (blank for hp41cv entries, populated for math1 entries) — which
changes the hp41cv matrix and requires a `docs-matrix-check` re-baseline; or (b) filter
the mixed 185-entry input into two output files — which requires a more complex renderer.
Option A's conditional emission is the minimum-blast-radius implementation (D-30.1
locked the binary-stays-1-in/1-out invariant).

**4. Future v3.1+ pacs scale linearly.**

Future XROM modules (Stat Pac, Time Pac, Advanced Matrix, Advantage) each add one JSON
file and one matrix file. The justfile gains one `cargo run` line per pac. The parity
test gains one test set per pac. No existing file is modified. This is the per-pac
pattern that scales linearly without re-rendering existing files or modifying existing
tests. Under Option B, each new pac would need to be merged into a growing monolithic
JSON, re-running all ~185+ entry validations on each addition.

---

## Consequences

### Positive

- Zero v2.2 regression risk: `hp41cv-functions.json`, `hp41cv-function-matrix.md`, and
  the v2.2 parity test are untouched by Phase 28 and Phase 29.
- Clean per-pac test surfaces: `function_matrix_parity.rs` tests are independent per
  file; a regression in Math Pac I does not affect hp41cv CI feedback.
- Matrix-renderer surgical edit (Plan 30-01) is the minimum-blast-radius implementation:
  one conditional column emission, one new justfile invocation line, zero changes to
  the renderer's 1-in/1-out signature (D-30.1).
- XROM-column conditional emission is a structural drift-catch: a `hp41-math1-functions.json`
  entry missing the `xrom` block renders an empty cell in the matrix — immediately visible
  to a human reviewer without running a test.

### Negative

- A reader looking for "all functions" must open two JSON files rather than one. The
  `?`-overlay in CLI + GUI loads both files at startup (Phase 29 CLI: second `OnceLock`;
  Phase 31 GUI: Vite parallel import). Navigation between the two is documented in the
  README soft-claim link to `docs/hp41-math1-function-matrix.md`.
- The `hp41-cli/src/help_data.rs` pattern grows from one `OnceLock` to two `OnceLock`
  instances. The lazy-init pattern is well-established and tested; the second instance
  follows identically.

### Neutral

- Future v3.1+ pacs (Stat Pac, Time Pac, Advanced Matrix, Advantage) inherit the
  separate-file convention. Each pac adds one JSON + one matrix without re-rendering
  existing files. If the pac count grows to 5+, an index file linking all matrix files
  could re-emerge (deferred idea from D-30.2 review); not actionable in v3.0.
- The `xrom: { module, module_id, function_id }` schema addition to the Math Pac I JSON
  is orthogonal to the separation decision — both options could carry the same XROM
  fields. The separation decision is about which file carries those fields, not about
  whether those fields exist.

## Alternatives Considered

### Option B: Combined `hp41cv-functions.json` Extended to ~185 Entries

The rejected alternative is described verbatim in 28-CONTEXT.md C-28.3 (locked 2026-05-16):

> **C-28.3 (ADR-005):** JSON-pipeline shape — separate `docs/hp41-math1-functions.json`
> file (sibling to `hp41cv-functions.json`), identical schema plus
> `xrom: { module, module_id, function_id }` object per entry.
>   - **Why:** zero migration churn on 130 existing v2.2 entries; cleaner per-test
>     scoping; aligns with future v3.1+ pacs each getting their own JSON.

**Additional context on why Option B was rejected:**

Re-validating 130 entries against a new schema (that adds an optional `xrom` field)
risks introducing schema-validation regressions in the v2.2 path that was stable. The
v2.2 `function_matrix_parity.rs` test would need to filter by `xrom.is_some()` to
separate the two test surfaces — adding conditional complexity to a test that was
previously straightforward. If the filter condition itself is wrong, a Math Pac I
regression could be masked by a passing hp41cv entry in the same test run.

The per-file OnceLock pattern also breaks under Option B: `help_data.rs` would need
to either (a) load one large JSON and runtime-filter by `xrom` presence — adding a
filter step to every `?`-overlay render — or (b) split the combined JSON into two
in-memory lists at load time, which is the same structural split as Option A but
done at runtime rather than at file-system level.

The D-25.17 hard-build-blocker discipline (malformed JSON panics) applies to both
options. Under Option B, any malformed `xrom` entry in a 185-entry file blocks the
entire build — 130 existing entries are held hostage by 55 new entries. Under Option A,
`hp41cv-functions.json` malformation blocks only the hp41cv path; `hp41-math1-functions.json`
malformation blocks only the Math Pac I path. Blast radius is proportionally smaller.
