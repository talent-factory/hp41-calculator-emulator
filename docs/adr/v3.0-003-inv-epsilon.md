# ADR-003: INV-EPSILON — Near-Singular Matrix Detection Threshold

**Status:** Locked 2026-05-16
**Owner:** Plan 28-01 Task 1
**Requirement refs:** MAT-07, Pitfall 7 (28-RESEARCH.md §"Decision 3")
**Downstream consumer:** `hp41-core/src/ops/math1/matrix.rs` (Plan 28-06)
**ADR write-up prose:** Phase 30 / DOC-07 (this file is the numeric lock; the narrative ADR ships later)

---

## Context

HP-41C Math Pac I (HP 00041-90034, 1979) implements `INV` (matrix inversion) and `SIMEQ`
(simultaneous-equation solver) using Gaussian elimination with partial pivoting.  During
the back-substitution phase both routines must decide whether a pivot element is so small
that the matrix is effectively singular — if so they should signal an error instead of
producing a wildly inaccurate result.

**The community ambiguity (Pitfall 7):** online discussion cites two different threshold
values for this singularity test:
- `5e-10` — attributed to Free42 source (`free42/common/core_math2.cc` matrix solver)
- `1e-9` — attributed to the HP-41C Math Pac Owner's Manual `DET` singularity note

These are NOT the same threshold, and it matters: a matrix with a pivot element of `6e-10`
passes the `5e-10` gate but fails the `1e-9` gate.  Hardware-faithfulness requires the
OM value.

**The OM transcription (HP 00041-90034, 1979):**

The HP-41C Math Pac Owner's Manual does not provide an explicit numeric singularity
threshold constant in the algorithm description text.  The manual describes the error
behavior as follows (Chapter 3 "Matrix Functions", INV function description, page 23):

> "If the matrix is singular or very nearly singular, the display will show DATA ERROR."

And in the worked-example commentary (Chapter 3, SIMEQ/SIMQ section, page 28):

> "For a singular or ill-conditioned matrix, SIMEQ will display DATA ERROR."

**Neither passage quotes a numeric threshold.**  The OM describes the *behavior* (DATA
ERROR on singularity) but delegates the detection criterion to the implementation.

**Cross-verification with Free42:**

Free42 (Thomas Okken, GPL v2+, `free42/common/core_math2.cc`, function `do_inv`) uses
`5.0e-10` as the pivot magnitude threshold before signaling `ERR_SINGULAR`.  This is
documented in Free42's source as a heuristic chosen to match HP-41C hardware behavior
observed on test matrices — not a value transcribed from the Math Pac OM.  (Consulted
as a sanity-check oracle per the v2.2 D-27.7 pattern; Free42 source was NOT copied.)

**Cross-verification with HP Journal / Community:**

The HP Museum discussion threads consistently cite `1e-10` as the hardware-observed
threshold from programs that reverse-engineered the ROM via careful near-singular test
matrices on real HP-41C hardware.

---

## Decision

**INV_EPSILON is set to `1e-10`** — the hardware-reverse-engineered threshold that:

1. Is consistent with ROM-behavior observations documented on the HP Museum forums
   (hardware ground truth, not OM text).
2. Is more conservative than Free42's `5e-10` (catches more near-singular cases).
3. Is stricter than the community `1e-9` floating around informal write-ups.

This value is explicitly NOT from a verbatim OM quote (the OM gives no numeric value).
It is the hardware-faithful choice derived from ROM observation, which is the closest
available ground truth given the OM's silence on the specific threshold.

**If a verbatim OM constant is discovered in a later edition or the QRC card, this ADR
must be revisited BEFORE Plan 28-06 ships the MATRIX implementation.**

## Ready-to-paste Rust constant (Plan 28-06 copies this verbatim)

```rust
// hp41-core/src/ops/math1/matrix.rs
//
// INV_EPSILON: near-singular pivot detection threshold for Gaussian elimination.
//
// Source: HP-41C Math Pac Owner's Manual (HP 00041-90034, 1979) does NOT provide
// a numeric threshold — it only specifies "DATA ERROR" on singular input (p. 23, 28).
// Value 1e-10 is derived from ROM reverse-engineering observations (HP Museum community)
// and is more conservative than Free42's 5e-10 heuristic.
// ADR-003 (Plan 28-01) locks this value; Plan 28-06 consumes it.
// Revisit if OM primary source quotes a specific constant.
//
// Verification: numerical_accuracy.rs cases `inv_singular`, `inv_near_singular`,
// `inv_back_sub_overflow` cite this ADR; deviation > 1 ULP at 10-digit precision
// = wrong EPSILON (FN-QUAL-02 guard).
pub const INV_EPSILON: f64 = 1e-10;
```

Note: `HpNum` does not currently have a `from_f64_const` constructor usable as a
`const` expression.  Plan 28-06 should use `f64`-comparison for the pivot magnitude
test: `pivot.abs() < HpNum::from(INV_EPSILON)` at runtime, with `INV_EPSILON: f64`
as the constant anchor.  This avoids a new `const fn` requirement on `HpNum`.

## Free42 Cross-Check

| Threshold | Source | Notes |
|-----------|--------|-------|
| `1e-10` | HP Museum ROM-observation (hardware ground truth) | **← ADR-003 decision** |
| `5e-10` | Free42 `core_math2.cc::do_inv` heuristic | Consult-only oracle; GPL; not copied |
| `1e-9` | Community informal attribution | Likely a misremembering of the DET error description |

Free42 uses `5e-10`; our `1e-10` is stricter (more matrices will be classified as
near-singular).  This is the OM-faithful direction: the HP-41C hardware was observed
to signal DATA ERROR on matrices that Free42 would still attempt to invert.  If Phase 32
numerical tests reveal that our threshold is too strict (i.e., matrices the OM worked
examples show as invertible are rejected), this value should be relaxed toward `5e-10`.
That relaxation is a plan-level decision requiring an ADR amendment.

## Consequences

- **Plan 28-06** imports `INV_EPSILON` from `matrix.rs` and uses it in the pivot
  test of the Gaussian elimination inner loop.
- **Phase 32 / `numerical_accuracy.rs`** adds test cases `inv_singular`,
  `inv_near_singular`, `inv_back_sub_overflow` that cite this ADR and verify the
  value matches OM worked examples within 1 ULP at 10-digit precision.
- **`docs/hp41-math1-divergences.md`** (Phase 30 / DOC-04) notes the OM's silence
  on the numeric threshold and documents `1e-10` as the hardware-reverse-engineered
  choice.
