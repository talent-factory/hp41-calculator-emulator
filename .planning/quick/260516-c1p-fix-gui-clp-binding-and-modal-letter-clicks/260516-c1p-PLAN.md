# Quick Task 260516-c1p — v2.2.1 GUI Parity Hotfix

## Goal

Close two GUI parity gaps discovered post-v2.2-ship that violate the
**CLI ↔ GUI parity invariant D-25.6** and contradict the verification
procedure in `docs/verifying-card-reader.md`:

1. **CLP has no GUI key binding** — Backend (`Op::Clp`, `key_map.rs::resolve`)
   and modal type (`pending_input.ts::'clp'`) are ready, but no key opens
   the modal and `MODAL_OPENERS` has no `clp_prompt` entry.
2. **On-screen-keyboard clicks cannot feed letters into LBL / XEQ / GTO /
   CLP / ASN-label modals** — `handleClick` forwards raw primary op-ids
   (`alpha_toggle`, `sigma_minus`, …) which the `isPrintableChar` predicate
   in `xeq_name`/`clp`/`assign_label` rejects (length > 1).

Both gaps were noted in `.planning/notes/v3.x-tech-debt-on-screen-modal-letter-input.md`
and originally scheduled for v3.x. The card-reader verification doc explicitly
claims `Mirror sections 3 and 4 exactly` for GUI runs and `Cross-UI guarantee`
on byte-identical hashes — so the gap is a documentation-breaking defect,
not just tech-debt.

## Design

### CLP binding — mode-aware `shifted` on √x

Hardware-faithful semantics (per HP-41C/CV Owner's Manual + 
`docs/hp41cv-functions.json` line 929 `"key_path": "f-C"`):

- **Outside PRGM mode:** `f + √x` → `Op::Sq` (x²). Unchanged.
- **Inside PRGM mode:** `f + √x` → opens `clp_prompt` modal. New.

Implementation: optional `shiftedInPrgm?: { id: string; label: string }`
field on `KeyDef`. Only the `sqrt` entry carries it. `handleClick`'s
resolution rule 3 (`shiftActive && key.shifted`) consults
`annunciators.prgm` and prefers `shiftedInPrgm.id` when set.

**Visible label on the key stays `x²`** — the real HP-41 doesn't relabel
in PRGM mode either; the mode-aware behavior is part of the operator's
muscle memory.

### Letter clicks in label modals — `alphaChar` fallback

When `pendingInput.kind` is one of `xeq_name` / `clp` / `assign_label`
AND the clicked key carries an `alphaChar`, `handleClick` overrides
`routedKey` to that single character before calling `handleModalKey`.
This lets the modal's existing `isPrintableChar(key) && acc.length < CAP`
predicate accept the click.

This is **Option A** from the backlog note. Smallest diff; per-modal-kind
gate keeps the behavior surgical (no regression risk for non-text modals).

### Doc patch — `docs/verifying-card-reader.md`

- Section 2 row 01: clarify the GUI flow (LBL modal accepts letters
  directly — no `ALPHA … ALPHA` wrap).
- Section 3 step 4: spell out the GUI key sequence for CLP
  (`PRGM mode → SHIFT + √x → type label → ENTER`).
- Section 6: mention the new mode-aware CLP binding so the cross-UI
  parity claim is empirically verifiable.

### Task 3: Mode-aware shifted resolution in `handleClick`

**Files:** `hp41-gui/src/App.tsx`

Modify the rule-3 branch (currently `else if (shiftActive && key.shifted)`):

```ts
} else if (shiftActive && (key.shifted || key.shiftedInPrgm)) {
  const prgmOn = calcState?.annunciators.prgm ?? false;
  if (prgmOn && key.shiftedInPrgm) {
    effectiveId = key.shiftedInPrgm.id;
  } else if (key.shifted) {
    effectiveId = key.shifted.id;
  } else {
    // shiftedInPrgm exists but we're not in PRGM — fall through to primary
    effectiveId = key.id;
  }
  consumesShift = true;
}
```

(For the √x entry the fall-through case can't actually trigger because
it has both `shifted` and `shiftedInPrgm`, but the guard keeps the rule
robust against future `shiftedInPrgm`-only entries.)

### Task 4: `alphaChar` fallback for label modals

**Files:** `hp41-gui/src/App.tsx`

Inside `handleClick`, after the existing rule-6 modal-routing block
constructs `routedKey` (the `if (pendingInput.kind === 'assign_key') { … }
else if (effectiveId === 'enter') { … } else if (effectiveId === 'clx_or_a')
{ … } else { routedKey = effectiveId; }` chain), insert ONE more branch
**before** the trailing else:

```ts
} else if (
  (pendingInput.kind === 'xeq_name'
   || pendingInput.kind === 'clp'
   || pendingInput.kind === 'assign_label')
  && key.alphaChar
) {
  routedKey = key.alphaChar;
}
```

The override fires only when the clicked key has an `alphaChar` (so SHIFT,
CHS, ENTER, ←, etc. still hit the default branch and remain no-ops or
control keys). `handleModalKey`'s `xeq_name` / `clp` / `assign_label` arms
already uppercase the char.
