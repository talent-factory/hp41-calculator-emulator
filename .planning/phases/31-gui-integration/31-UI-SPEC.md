# Phase 31 — UI Design Contract

> Visual and interaction contract for `hp41-gui` Math Pac I integration. Generated by gsd-ui-researcher from CONTEXT.md (D-31.1..D-31.14) + ROADMAP Phase 31 + existing `hp41-gui/src/App.css` (366-line vanilla-CSS design system, frozen since v2.0/v2.1/v2.2 ship).
>
> **Phase scope is additive, not redesign.** v2.0–v2.2 already established the HP-41C visual language (authentic SVG skin, dark calculator chassis, 14-segment LCD, 5×8 keyboard, annunciator row, print panel, program panel, toast overlay, `?`-help overlay). Phase 31 extends that vocabulary with: (a) LCD modal-prompt alternation, (b) `?`-overlay two-section grouping, (c) hardware-faithful CAT 2 PSE scroll, (d) cancellation-trigger routing on R/S + Esc. **No new colors. No new fonts. No new spacing tokens. One new TypeScript constant. One new CSS-`≡` glyph verification.**

**Why no design-system tool:** the HP-41C is a 1979 hardware artefact with a fixed visual identity (`#0d0d0d` chassis, `#c8e6c9` LCD green, `#f5a423` HP orange, `#e8e8c0` annunciator yellow). shadcn / Tailwind / Material would all be inappropriate dependencies; vanilla CSS in `App.css` is the locked design substrate.

**shadcn gate skipped — rationale documented:** D-10 (Phase 15) explicitly chose vanilla CSS for hardware fidelity. Re-asking would relitigate a settled architectural decision spanning 18 phases.

## State & Interaction Contract

Phase 31 introduces state-machine extensions that downstream gsd-planner / gsd-executor must implement verbatim.

### R/S key (id `r_s`) — 3-way state-routed (D-31.1)

```
on click(r_s) OR on keyDown(matching r_s binding):
  if busyRef.current === true: return  // debounce
  if pending_input !== null: route to pending_input handler (D-07 above)
  if state.modal_program is not None:
    invoke('submit_modal')
  else if state.is_running:
    invoke('request_cancel')
  else:
    invoke('run_stop')  // existing v2.1 behavior
```

### Esc key — universal cancel (D-31.2)

```
on keyDown(Escape):
  if state.modal_program is not None:
    invoke('cancel_modal')
  else if state.is_running:
    invoke('request_cancel')
  else if shiftActive:
    setShiftActive(false)  // existing v2.1 behavior
  else:
    no-op
```

### Post-dispatch auto-open (D-29.9 mirror)

```
useEffect after every get_state response:
  if state.modal_program is not None
    AND modal_program.requires_alpha_label === true   // via CalcStateView-exposed boolean
    AND pending_input === null:
      setPendingInput({ type: 'xeq_by_name', acc: '', mode: 'collect-for-modal' })
```

### LCD-alternation routing (D-31.5)

```
commands.rs::handle_get_state computes state.display:
  if state.modal_program is not None AND state.entry_buf is empty:
    state.display = truncate_with_continuation(
                      state.modal_prompt.as_deref().unwrap_or(""),
                      12,        // LCD width
                      '\u{2261}' // HP-41 ≡ continuation marker
                    )
  else:
    state.display = (existing v2.2 logic — entry_buf live, or formatted X register)
```

`truncate_with_continuation(s, n, marker)` semantics:

- if `s.chars().count() <= n` → return `s` verbatim
- else → return `s.chars().take(n - 1).collect::<String>() + marker.to_string()` (first n-1 chars + marker as final char)

Edge cases tested: 12-char prompt (no truncation), 13-char (truncated, marker at position 12), 14-char (`FUNCTION NAM≡`), Unicode-rich prompt (per-char not per-byte truncation), empty `modal_prompt` (renders as empty 12-char display = LCD blank).

### Cancellation feedback (D-31.3)

```
when op_integ / op_solve / op_difeq returns HpError::Canceled:
  → existing From<HpError> for GuiError conversion
  → existing toastMsg state set to "CANCELED"
  → existing 2s setTimeout dismiss
  → existing .toast CSS animation
```

No new render code. No new error pathway. Cancellation feedback inherits the entire v2.1 stub-error toast pattern.

### CAT 2 PSE scroll (D-31.12, D-31.13, D-31.14)

```
Op::Catalog(2) body (verify in research-prep — full body vs stub from Phase 28):
  if state.print_buffer queue is empty:
    push "XROM Math 1" line
    for each (name, _op) in MATH_1.ops in registration order:
      push name line
    interleave with v2.2 PSE-step infrastructure (~500ms per line)
  on R/S during scroll: pause/resume (existing v2.2 CAT 1 behavior)
  on any other key during scroll: cancel scroll (existing v2.2 CAT 1 behavior)
```

Phase 31 frontend changes: **none beyond the existing v2.2 print-panel auto-scroll behavior**. Lines arrive via the existing `print_buffer` drain on every `get_state` response.

### `?`-overlay parallel-load (D-31.10)

```

  import math1Json from '@docs/hp41-math1-functions.json'; // NEW Phase 31
```

## Accessibility

Phase 31 explicitly accepts the hardware-fidelity trade-off (D-31.7) over WCAG-style fallback paths. Documented for downstream agents:

- **LCD-prompt truncation has NO accessibility fallback** (D-31.7) — no tooltip, no print-panel header, no ARIA-label. Users learn full prompt text from `?`-overlay descriptions.
- **`is_running` annunciator is the sole busy/cancellable indicator** (D-31.4) — no inline hint text, no R/S tooltip. Matches real HP-41 BUSY annunciator UX.
- **Toast cancellation message** ("CANCELED") inherits the existing `.toast` `role` attribute pattern from v2.1; `role="status"` already applied in `App.tsx` per v2.1 line documentation.
- **Help-overlay collapsible sections** (D-31.8) — section headers should be focusable `<button>` elements with `aria-expanded` attribute reflecting open/closed state; planner instructs accordingly. Both sections open by default = best discoverability without requiring screen-reader users to expand them.
- **`request_cancel` Tauri command** is keyboard-only (R/S + Esc); no mouse-only escape hatch. Mouse-only users discover R/S via the on-screen keyboard (the keycap is clickable).
