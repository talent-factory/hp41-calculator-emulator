**Step 5: Create hp41-gui/src-tauri/src/lib.rs**

Exact content (from PATTERNS.md — empty Tauri shell, Phase 13 scope only):

```rust
#![deny(clippy::unwrap_used)]

use std::sync::Mutex;

pub type AppState = Mutex<hp41_core::CalcState>;

#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .setup(|app| {
            app.manage(Mutex::new(hp41_core::CalcState::new()));
            Ok(())
        })
        .invoke_handler(tauri::generate_handler![
            // Tauri commands registered here in Phase 14
        ])
        .run(tauri::generate_context!())
        .expect("error while running tauri application")
}
```

IMPORTANT: `AppState` type alias is defined here even though Phase 13 has no commands yet.
It establishes the correct `Mutex<CalcState>` pattern so Phase 14 gets a compile error (not
a runtime panic) if State<> types are wrong (PITFALLS.md #5).

The `#![deny(clippy::unwrap_used)]` lint applies to lib.rs (where production code lives).
The `.expect("error while running tauri application")` on the Builder is acceptable because
Tauri panic on startup is unrecoverable — `.expect()` with a message is correct here.

Mutex poisoning in command handlers (Phase 14) MUST use `.unwrap_or_else(|e| e.into_inner())`
not `.unwrap()` or `.expect()` — document this constraint for Phase 14 (PITFALLS.md #7).
