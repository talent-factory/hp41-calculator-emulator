<objective>
Graduate the existing-but-vacuous `math1_op_test_count.rs` and `xrom_shadowing.rs` meta-gates to non-vacuous, ship a new `lint_math1_assertions.rs` assertion-discipline lint, close any per-file `ops/math1/*.rs` coverage gaps surfaced by `just coverage`, and optionally add the two missing QUAL-08 user-callback regression test categories (GTO-out + recursion-cap).
</objective>

<must_haves>
- `just coverage` reports `hp41-core` line coverage ≥ 95.0 % AND regions ≥ 93.0 % (D-27.2 — held at v2.2 level, NOT raised); per-`ops/math1/*.rs` file ≥ 90 %.
- `cargo test -p hp41-core --test math1_op_test_count` reports `test result: ok. 1 passed; 0 failed` with the gate iterating ALL 45 Math Pac I `Op` variants in `MATH_1.ops` (no vacuous early-return) per Pitfall 16 / D-27.3.
- `cargo test -p hp41-core --test xrom_shadowing` reports `test result: ok. N passed; 0 failed` with the gate iterating all 52 entries in `MATH_1.ops` against the 18-entry `BUILTIN_CARD_OP_NAMES` allowlist (no Math Pac I name shadows a v2.2 builtin) per Pitfall 1.
- `cargo test -p hp41-core --test lint_math1_assertions` reports `test result: ok. 2 passed; 0 failed` covering (a) no `assert_eq!(decimal, decimal)` on iterated results in `tests/math1_*.rs`, and (b) no manual-tolerance pattern `(a - b).abs() < EPSILON` in `tests/math1_*.rs` per Pitfall 14 + Pitfall 17.
- `cargo test -p hp41-core --test math1_user_callback` continues to pass with ≥ 5 distinct QUAL-08 categories explicitly named (nested-rejection, STO-clobber, STOP-during-INTG, GTO-out, recursion-cap).
</must_haves>

<threat_model>
**T-32-04: Coverage gate gaming (severity: medium)** — Meta-tests in this plan (`math1_op_test_count.rs`, `xrom_shadowing.rs`, `lint_math1_assertions.rs`) could be silently weakened in a future commit (e.g., reverting the vacuous early-return removal, allowlisting new offenders without rationale, or weakening the line-mention heuristic). Mitigation owned by this plan:
  - The `// Catches: <bug class>` doc-comment on each new `#[test]` names the bug class explicitly (D-27.1 discipline) — reviewers can spot weakening.
  - The per-Op count baseline (minimum: TriSaa=6, TriSas=6 per RESEARCH.md Per-Op Test Count Audit) is captured verbatim in the doc-comment of `math1_op_test_count.rs` so a drop below the baseline is immediately visible in diffs.
  - The 5 known manual-tolerance offenders in `math1_matrix.rs:298,:323,:360,:427,:431` are the lint's reference truth at land time; any allowlisting is an explicit `// LINT-EXEMPT: <reason>` annotation reviewed in PR.

No `high` severity threats — test-only phase.
</threat_model>
