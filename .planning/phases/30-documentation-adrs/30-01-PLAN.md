Conditional column emission rule (D-30.3): if ANY entry in the input file has `xrom != None`, render the XROM column header + per-row cells. If ALL entries have `xrom == None` (the hp41cv case), do NOT emit the column. This keeps `docs/hp41cv-function-matrix.md` byte-identical to its HEAD state.

XROM cell rendering format: `Math 1 / 7-{function_id}` (e.g. `Math 1 / 7-1`, `Math 1 / 7-2`, …). Module name comes from `xrom.module`; the slash separates module-name from `module_id-function_id`.

Column position: between `Display` and `Category` per D-30.3.

  <action>
Make these surgical edits to `scripts/docs-matrix/src/main.rs` (the only file in the binary):

1. **Add an `XromRef` struct** (under the existing `Entry` declaration) deriving `Deserialize`, with three fields matching the JSON shape: `module: String`, `module_id: u32`, `function_id: u32`. Add `#[allow(dead_code)]` on the struct (Rust will warn about unused fields; the renderer only reads `module` and `function_id` — `module_id` is parsed for schema-validation completeness).

2. **Extend the `Entry` struct** with an additional field: `xrom: Option<XromRef>` carrying `#[serde(default)]`. The `#[serde(default)]` makes the field optional on inputs that don't have it (the hp41cv JSON), and parses the object when present (the math1 JSON). This is the only struct widening in this plan and the documented surgical exception to D-30.1.

3. **Modify `render_table`** to render an optional XROM column. The column appears IFF any entry in the slice has `xrom.is_some()` (use `entries.iter().any(|e| e.xrom.is_some())` to compute a bool `has_xrom` at the top of `render_table`). When `has_xrom` is true:
   - The header line becomes `"| Op | Display | XROM | Category | Status | Phase | Key Path | Description |\n"` (insert `XROM` between `Display` and `Category`).
   - The separator line becomes `"|----|---------|------|----------|--------|-------|----------|-------------|\n"` (match column count).
   - Each row inserts a cell `Math 1 / 7-{function_id}` after the `display_name` cell, using `e.xrom.as_ref().map(|x| format!("{} / {}-{}", x.module, x.module_id, x.function_id)).unwrap_or_else(|| "—".to_string())` — the fallback `"—"` handles mixed inputs gracefully even though the production data flow won't have any.

   When `has_xrom` is false (hp41cv case), the existing header/separator/row format remains EXACTLY as today — byte-identical output is the D-30.2 invariant.

4. **Modify `render_markdown`** to use a generic top-of-file title. Replace the hardcoded `"# HP-41CV ROM Function Matrix\n\n"` and the hardcoded path string `"`docs/hp41cv-functions.json`"` with a derivation that:
   - Reads the basename of the input JSON path (passed in via `main` — pass `json_path` into `render_markdown` as a new parameter).
   - If basename ends with `hp41cv-functions.json` → title is `"# HP-41CV ROM Function Matrix"` and the "Generated from" path string is `"`docs/hp41cv-functions.json`"` (preserve current text exactly).
   - If basename ends with `hp41-math1-functions.json` → title is `"# HP-41C Math Pac I Function Matrix"` and the "Generated from" path string is `"`docs/hp41-math1-functions.json`"`.
   - Anything else → fall back to a generic `"# Function Matrix"` (defensive; not exercised by current inputs).

   Add a comment block above the function clarifying that this string-match dispatch is intentional and avoids the renderer needing a new CLI argument for the title (D-30.1 keeps the binary signature 1-in/1-out).

5. Keep `escape_pipe` and the rest of the file unchanged.

Do NOT touch `scripts/docs-matrix/Cargo.toml`, `scripts/docs-matrix/Cargo.lock`, or any other file in this task. Zero new dependencies — `serde` already supports `Option<T>` and nested struct deserialization out of the box.

Rationale for this widening (per D-30.1 / D-30.3 reconciliation): the CONTEXT.md states the binary keeps its 1-in/1-out signature; nothing in CONTEXT.md prohibits widening the Entry struct schema to accept an optional field. The struct widening is the minimum-blast-radius alternative to (a) keeping the binary frozen and post-processing the matrix file (which would add a second tool to maintain), or (b) widening the CLI to accept a title argument (which adds argument-parsing surface). D-30.3 mandates the XROM column for the Math Pac I matrix specifically — the conditional emission keyed off `xrom.is_some()` is the cleanest implementation.
