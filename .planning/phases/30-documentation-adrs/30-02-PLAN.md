## 1. OM Divergences

(Numerical / behavioral mismatches with OM-quoted examples.)

### D-30-01: User-Program Scratch Register Clobber During INTG/SOLVE/DIFEQ
- **OM citation**: HP 00041-90034 (1979), Chapter 3 "Numerical Integration", p. 35 — "Important: The INTG program uses registers R00 through R07. Do not use these registers in your user function while INTG is active, or the integration result will be incorrect."
- **Our behavior**: The emulator does NOT snapshot/restore R00–R07 around the user-program callback. If the user's LBL function executes `STO 00..07`, the solver's internal state corrupts and the result is wrong. No error is raised — the result is silently incorrect.
- **OM behavior**: Identical to ours. The OM warns the user but does not have the calculator detect or prevent the corruption. R00–R07 are documented scratch registers for INTG/SOLVE/DIFEQ; managing register usage is a user responsibility.
- **Rationale**: Hardware-faithful behavior (Math Pac I behaves this way on real hardware). Mitigation options (snapshot/restore, error-on-STO-in-callback) were considered and rejected: snapshot/restore diverges from hardware behavior and adds overhead; STO detection cannot be done without significant overhead and would reject valid user programs. Documentation in this file is the accepted mitigation.
- **See**: `hp41-core/tests/math1_user_callback.rs::user_fn_stores_to_scratch_corrupts_integ`; ADR-002 (`v3.0-002-user-callback-policy.md`); 28-CONTEXT.md C-28.2.

### D-30-02: POLY Complex-Root Multiplicity Rendered as Cluster
- **OM citation**: HP 00041-90034 (1979), Chapter 4 "Polynomial Roots", POLY worked example (planner transcribes the exact page during the entry write — likely Ch. 4 §POLY/ROOTS).
- **Our behavior**: When POLY finds a multiplicity-k root (e.g. `(x-1)^3 = 0` has root 1 with multiplicity 3), our implementation returns the root as a cluster of k nearby points rather than as a single point marked with its multiplicity. This is the Pitfall 5 mitigation locked in the v3.0 architecture: multiplicity-as-cluster is the convention adopted for the modified-Bairstow output format.
- **OM behavior**: The OM does not explicitly state multiplicity-as-cluster vs single-root-with-multiplicity convention; the modified-Bairstow algorithm naturally produces clustered output, so this divergence is a documentation gap in the OM, not an algorithmic divergence. Our cluster output matches Free42's observed output on identical inputs (sanity-check oracle, not copied).
- **Rationale**: Modified-Bairstow's clustering is hardware-faithful (the iterative refinement converges to nearby points, not to a single point with annotated multiplicity). Adding a "multiplicity counter" post-processing step would diverge from the algorithm's natural output. Documentation in this entry + a `// Pitfall 5` comment in `ops/math1/poly.rs` is the chosen mitigation.
- **See**: Pitfall 5 (28-RESEARCH.md); `hp41-core/src/ops/math1/poly.rs`; 28-CONTEXT.md POLY discussion.

### D-30-03: INTG Convergence Threshold Tied to DisplayMode
- **OM citation**: HP 00041-90034 (1979), Chapter 3 "Numerical Integration", pp. 35–36 — "The program halts when two successive approximations agree to the number of digits displayed (as set by FIX, SCI, or ENG)" and the more precise "½ ULP of displayed precision" footnote on p. 36.
- **Our behavior**: `integ_threshold(mode) = 5 × 10^(-(decimals + 1))` where `decimals` is the FIX/SCI/ENG digit count from `state.display_mode`. ADR-004 documents the lock. Different DisplayMode settings produce different convergence outcomes for the same integral — by design and by OM specification.
- **OM behavior**: Identical formula derivation per the OM's ½-ULP wording.
- **Rationale**: Hardware-faithful per OM. The DisplayMode tie-in is THE defining feature of HP-41C INTG's convergence behavior; emulator-without-this-tie would be wrong.
- **See**: ADR-004 (`v3.0-004-intg-threshold.md`); Pitfall 2 (28-RESEARCH.md); `hp41-core/src/ops/math1/integ.rs`; `hp41-core/tests/numerical_accuracy.rs` Pitfall-2 detector cases.

### D-30-04: FACT Integer-Only — No GAMMA Extension
- **OM citation**: HP 00041-90034 (1979) Math Pac OM does NOT add a Gamma function (Γ for non-integer factorial); FACT in the base HP-41CV ROM caps at X ≤ 69 integer-only per the HP-41C/CV QRG.
- **Our behavior**: FACT (carried over from v2.2 Phase 27) accepts integer X only; X in `27..=69` returns `Overflow` (effective cap is X ≤ 26 due to the `Decimal::from_f64` overflow wall, calibrated by Phase 27 proptest); X ≥ 70 returns `OutOfRange`. We do NOT add a `GAMMA` function for non-integer-factorial — the Math Pac I OM does not contain GAMMA either (the related HP-15C "FACT" extension to gamma is a different model's feature).
- **OM behavior**: Identical — FACT integer-only on both HP-41C/CV and Math Pac I.
- **Rationale**: Scope discipline. Adding GAMMA would mean implementing the Lanczos approximation or similar (~100 lines + a new dev-dep or hand-rolled coefficients); the scope is "Math Pac I behavioral emulation", not "calculator math superset". GAMMA is captured as a deferred idea (Phase 32+ — if a future user explicitly asks for it).
- **See**: README.md `## Documented Divergences from HP-41 Hardware` (v2.2 FACT line); `hp41-core/src/ops/math.rs::op_fact`; v2.2 Phase 27 D-27.5 (FACT case citations).

## 2. Emulator Extensions

(Functions/behaviors we added that aren't in OM 1979.)

### D-30-05: XEQ "REAL" — Deactivates `complex_mode`
- **OM citation**: N/A — emulator extension. Not present in HP 00041-90034 (1979).
- **Our behavior**: `XEQ "REAL"` sets `state.complex_mode = false`, restoring the four-level stack to standard X/Y/Z/T semantics. No other side effects on the stack contents. Reachable from CLI + GUI via the XEQ-by-name modal.
- **OM behavior**: The OM provides no mechanism to deactivate complex mode (the user toggles complex mode via the `Op::ToggleComplex` op — analogous to the OM's complex-mode entry point — but provides no explicit "exit complex mode" function). Real Math Pac I hardware relies on the user pressing a complex-stack op to maintain mode; mode persistence is implicit.
- **Rationale**: UX divergence locked in D-28.3 (28-CONTEXT.md). Without an explicit deactivation entry point, a user who entered complex mode via XEQ has no clean exit path other than restarting the calculator state. The XEQ "REAL" extension provides one. It is grep-documented in `hp41-core/src/ops/math1/xrom.rs` and visible in the CLI ?-overlay + GUI ?-overlay (Math 1 Pac section).
- **See**: 28-CONTEXT.md D-28.3; REQUIREMENTS.md CMPLX-18; `hp41-core/src/ops/math1/xrom.rs`.

## 3. Behavioral Policies

(Cross-cutting rules that aren't strictly divergences but are decisions worth catalog-citing.)

### D-30-06: Strict-Reject Nested INTG/SOLVE/DIFEQ
- **OM citation**: HP 00041-90034 (1979), Chapter 3 "Numerical Integration" describes INTG as non-reentrant — the user is warned NOT to call INTG from inside the user function. Math Pac I OM does not state an explicit error behavior for the nested case; on real hardware, nested calls produce undefined behavior (stack corruption, wrong answers, sometimes hangs).
- **Our behavior**: `op_integ` / `op_solve` / `op_difeq` detect nested invocation at op entry (the outer `integ_state` / `solve_state` field is `Some(...)`) and return `HpError::InvalidOp` with the message "NESTED INTG/SOLVE NOT ALLOWED". The user-program callback sees the error and aborts cleanly. No state corruption.
- **OM behavior**: Implementation-defined "undefined behavior" on real hardware. Our strict-reject is a stricter, safer interpretation of the OM's prohibition.
- **Rationale**: Locked in ADR-002 (28-CONTEXT.md C-28.2). The OM warns against the nested case; we enforce the warning at the engine level rather than relying on the user to read the manual. This is one of the few cases where emulator behavior is more user-friendly than real hardware — and it's documented as such.
- **See**: ADR-002 (`v3.0-002-user-callback-policy.md`); 28-CONTEXT.md C-28.2; XROM-08 (REQUIREMENTS.md); `hp41-core/tests/math1_user_callback.rs`.

### D-30-07: Modal R/S Submits Numeric Input
- **OM citation**: HP 00041-90034 (1979), p. 13 — "Press R/S to continue" in the context of the modal prompt sequences (MATRIX `ORDER=?`, INTG `FUNCTION NAME?`, etc.).
- **Our behavior**: Inside a `ModalProgram` flow (e.g. MATRIX entering element values, INTG accepting `A=?` / `B=?` / `N=?` / `FUNCTION NAME?`), pressing R/S submits the current `entry_buf` value as the response to the active prompt, advancing the state machine. Locked in D-28.5.
- **OM behavior**: Identical per OM p. 13.
- **Rationale**: Hardware-faithful. R/S has dual semantics on real HP-41C — "run/stop" outside a modal, "submit input" inside a modal. The dual behavior is preserved by routing the R/S key through `pending_input` first (D-25.11 invariant from v2.2).
- **See**: 28-CONTEXT.md D-28.5; `hp41-cli/src/app.rs::handle_key` R/S routing; `hp41-gui/src-tauri/src/commands.rs::run_stop` Tauri command.

Sections:

1. `## Context` — Describe the problem: Math Pac I adds ~40 new functions. Two strategies exist:
   - **Option A** (chosen): one `Op` variant per function (`Op::Sinh`, `Op::Cosh`, `Op::Matrix`, `Op::Integ`, ...).
   - **Option B** (rejected): a single `Op::XromCall(u16)` variant that table-dispatches by module_id/function_id.

   Explain the 4-exhaustive-match invariant: every `Op` variant must land in `dispatch()` + `execute_op()` + CLI prgm_display + GUI prgm_display. Option B collapses this to one arm; Option A keeps the explicit per-function arms.

2. `## Decision` — State the lock: Option A. Then justify in 2–3 paragraphs:
   - Preserves the compile-time exhaustive-match invariant (the 4-way drift catch that has prevented bugs in v1.0–v2.2 — cite the CLAUDE.md "Every new Op variant must land in 4 places" pattern).
   - Per-Op test count ≥ 5 (Pitfall 16) maps naturally to ~40 new variants × 5 = 200 new tests; Option B would require a single super-test with table-driven cases — harder to bisect failures.
   - CATALOG-2 in Phase 31 GUI (the upcoming module listing) is easier to derive from a static `&[XromModule]` table where each entry references its `Op::*` directly, vs Option B's runtime function_id lookup table.

3. `## Consequences` — three subsections:
   - **Positive**: explicit per-Op tests + bisectable failures + compile-time exhaustive-match drift catch + simpler `op_display_name` arms.
   - **Negative**: ~40 new `Op` variants grow the enum size; criterion `bench/dispatch_overhead.rs` floor is `< 200 ns/op` (v2.2 baseline 65 ns) — Phase 32 / QUAL-08 must verify the floor holds.
   - **Neutral**: the resolver chain (`xrom_resolve` fires LAST per Pitfall 1) doesn't care which strategy is chosen — it returns an `Op` either way.

4. `## Alternatives Considered`:
   - **Option B: `Op::XromCall(u16)` table dispatch** — quote 28-CONTEXT.md C-28.1's rejected-option block verbatim per D-30.7:
     > [Quote the relevant block from `.planning/phases/28-xrom-framework-math-pac-i-core-ops/28-CONTEXT.md` lines 42–43 — the C-28.1 description of Option B. Read the file to find the exact wording. **Per W5 refinement (D-30.7 sourcing discipline):** if no standalone sentence delineates the rejected option, quote the full C-XX rationale paragraph verbatim in this blockquote and add a per-rejected-option paraphrase below it in an **Additional context** subsection per D-30.7.]
     Why rejected: the 4-way exhaustive-match invariant is the single most reliable bug-catching mechanism in the v1.0–v2.2 codebase; collapsing it loses the catch. The runtime function_id lookup adds branch-prediction noise to dispatch; the per-Op tests fan out to 40+ test functions vs one mega-test.
   - **Additional context**: Reference the v2.2 D-26 GUI parity invariant (CLI ↔ GUI must resolve identical strings to identical `Op`s); Option A makes parity trivially demonstrable via grep, Option B requires both surfaces to share an identical lookup table.
