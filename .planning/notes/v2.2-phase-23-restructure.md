## Decision Captured: Data Model

**Locked: B-full — Tagged `HpValue` in stack AND registers.**

```rust
// hp41-core/src/value.rs (NEW)
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
pub enum HpValue {
    Numeric(HpNum),
    Alpha([u8; 6]),
}
```

Stack slots (`x`, `y`, `z`, `t`, `lastx`) and `regs[i]` and the hidden registers
(`reg_m`, `reg_n`, `reg_o`) all carry the tag. Binary/unary numeric ops reject
`Alpha` with `HpError::AlphaData` (existing variant — reused from card-reader
path; new variant `NonNumeric` is an alternative if `AlphaData` semantics
collide).

**Rejected:** B-pragmatic (tag only registers, stack stays `HpNum`-only).
Reason: Synthetic-programming `ARCL X` / `POSA` workflows require an
alpha-tagged stack slot. User explicitly prioritized hardware fidelity over
refactor cost.

**Rejected:** Option A (parallel `text_regs` map) — abandons HP-41 "one slot
per register" semantics. Same hardware-fidelity argument.

**Rejected:** Option C (encode tag in `Decimal` mantissa) — `rust_decimal`'s
internal scale field is `0..=28`; using `0xFF` as marker would exploit an
implementation detail that breaks on minor version bumps. JSON persistence
loses the marker. Fails any review.

## Implication: New Phase 23 = HpValue Type-System Refactor

**Goal (draft):** Replace `HpNum` with tagged `HpValue` throughout
`hp41-core::stack::Stack`, `CalcState::regs`, `CalcState::reg_m/n/o`, and
`CalcState::Stack::lastx`. All ~130 existing ops type-check their stack/reg
reads against `Numeric` and return `HpError::AlphaData` (or a new
`HpError::NonNumeric` — TBD in discuss) for `Alpha` arguments. v1.0–v2.1
autosave JSON loads via a custom-Deserialize migration layer that wraps
plain `HpNum` strings as `HpValue::Numeric(_)`.

**Phase 23 has NO new FN-* requirements.** It is a structural phase
(precedent: Phase 8 "Tech Debt Cleanup"). Verification target:

1. All 660 existing tests stay green.
2. New migration test loads fixtures from
   `.planning/fixtures/autosave-v1.0.json`,
   `.planning/fixtures/autosave-v1.1.json`,
   `.planning/fixtures/autosave-v2.0.json`,
   `.planning/fixtures/autosave-v2.1.json`
   (must be captured BEFORE the schema change — see open work below).
3. Performance: key-press latency stays ≤ 50 ms median (current ~65 ns/op).
   The added `match HpValue::Numeric(x)` per stack read should be branch-
   predicted away by LLVM; budget for ≤ 5 % regression before raising
   alarm.
4. `#![deny(clippy::unwrap_used)]` stays clean — every `else` arm must
   return `Err(_)`, never `unreachable!()` or `.unwrap()`.

**Open gray areas (for the new Phase 23 discuss-phase):**

1. **Error variant choice** — reuse `HpError::AlphaData` (currently meaning
   "card-reader saw empty ALPHA") or add `HpError::NonNumeric` (cleaner
   semantics, but expands the public error surface). Card-reader code might
   need its own variant if we repurpose `AlphaData`.

2. **`HpValue` API surface** — helper methods on `HpValue`: `as_numeric()
   -> Result<HpNum, HpError>`, `unwrap_numeric()` (test-only), `is_alpha()`,
   `default() -> HpValue::Numeric(HpNum::zero())`. Decide which are public
   vs `pub(crate)`.

3. **Display of Alpha-in-X (TUI + GUI)** — what does the calculator
   "display" show when `stack.x = Alpha(b"GOODBY")`? HP-41 hardware shows
   the 6 chars right-justified (or left, TBD — needs hardware reference
   check). `format_hpnum` becomes `format_hpvalue` and handles both arms.
   `format_entry_buf_display` and `get_display_string` need parallel
   updates.

4. **Save-file migration strategy** — custom `Deserialize` impl on
   `CalcState` (whole-struct), or field-by-field `#[serde(deserialize_with
   = "...")]`? Test fixture set must include: empty stack, full stack with
   negative values, regs with non-zero entries, alpha_reg with multi-byte
   UTF-8 (verifies we're not accidentally byte-slicing).

5. **Save-fixture capture** — BEFORE the schema change lands, dump
   golden autosave files from current `develop` (v2.1 + v2.2-WIP at end of
   Phase 22) into `.planning/fixtures/`. This is a Wave-0 task.

6. **LASTX tag preservation** — when a binary op consumes an `Alpha`-tagged
   stack slot (now → error), does `LASTX` capture the previous X *before*
   the error? Real HP-41: LASTX always captures before the op fires. Our
   convention should match — and we need a test for it.

7. **`HpValue::default()` semantics** — `Default` derive vs manual impl.
   `CalcState::new()` initializes all 100 regs to zero — should that be
   `HpValue::Numeric(HpNum::zero())` (preserve existing behavior) or
   conceptually-cleaner `HpValue::Numeric(HpNum::default())`?

8. **`#[serde(default)]` cascade** — every new field gets it; every
   refactored field needs careful audit. The migration deserializer is the
   highest-risk addition.

9. **Performance sanity-check methodology** — Criterion bench before/after,
   or just `cargo bench --bench dispatch`? Phase 23's verification should
   include a numeric regression check.

10. **CHS / arithmetic on Alpha** — what should `CHS` do when X is Alpha?
    HP-41 hardware: silently no-ops (CHS only affects numeric). Decide:
    error (`HpError::AlphaData`) or silent no-op (matches HP-41).

## Implication: New Phase 24 = ALPHA Operations (formerly Phase 23)

**Scope is unchanged from current ROADMAP Phase 23** — same 6 ops, same FN-
requirements (FN-ALPHA-01..06), same success criteria. Only the phase
number and the underlying data model assumption change. Plans now layer on
top of `HpValue::Alpha`.

**Already-locked from this session (for new Phase 24 CONTEXT.md):**

- ASTO reads first 6 chars from `alpha_reg` (UTF-8 aware: `chars().take(6)`,
  but the 6-byte payload uses HP-41 ASCII codes — see open gray-area #3
  below). Writes `HpValue::Alpha(bytes)` to `regs[nn]`.
- ARCL of `HpValue::Numeric(n)` → `format_hpnum(&n, &display_mode)` then
  `alpha_reg.push_str(...)`.
- ARCL of `HpValue::Alpha(bytes)` → decode bytes back to chars, append to
  `alpha_reg` (subject to 24-char cap, silent truncation per existing
  `op_alpha_append` precedent).
- AROT reads N from X (HP-41 hardware — ROADMAP cross-cutting locked).

**Open gray areas (for the new Phase 24 discuss-phase — were originally
selected in this session but not yet discussed):**

1. **POSA search-arg convention** — now that the stack is tagged, POSA can
   simply require `state.stack.x` to be `HpValue::Alpha(needle_bytes)` and
   reject `Numeric` with `HpError::AlphaData`. This is the clean
   hardware-faithful answer.

   - Edge: empty needle (all-zeros bytes after trailing-zero strip) →
     return 0 (HP-41 hardware: position 0 is "match at start")? Or -1?
   - Edge: position indexing — 0-based per ROADMAP SC-5 ("QUICK at position
     4 in 'THE QUICK BROWN FOX'") — confirm.

2. **ATOX character set + consumption** — does ATOX consume (remove) the
   first ALPHA char after extracting its code? HP-41 hardware: YES,
   consumes. Code-set: HP-41 extended (0..255 including ∡, π, ◊) vs pure
   ASCII (0..127) vs Unicode codepoint (Rust `char as u32`).

   - Recommendation (for discuss): use HP-41 extended set with an
     explicit lookup table in `hp41-core/src/alpha_charset.rs`. The few
     non-ASCII HP chars (∡, π, ⊢, ◊, Σ, ‾, È, etc.) are HP-defined and
     constant. ATOX of an unknown char (Unicode codepoint > 255) → error
     `HpError::InvalidInput`.

3. **XTOA validation** — X must be `HpValue::Numeric` and `[0, 255]` and
   integer. Non-integer? Round-to-nearest (HP-41 hardware) or error? Out
   of range? HP-41: wraps mod 256 silently. We have `#![deny(unwrap)]`
   and a zero-panic invariant, so either path works — pick one and lock.

4. **AROT edge cases** —
   - N=0 → no-op (locked).
   - |N| ≥ len → modulo by `chars().count()`.
   - Empty ALPHA → no-op (no error).
   - Non-integer X → truncate (HP-41) or error (cleaner)? Recommend:
     truncate to match HP-41.

5. **POSA edge cases** — empty haystack, empty needle, needle longer than
   haystack. Lock against HP-41 hardware manual.

6. **Lift effects per op** — all 6 ALPHA ops are `LiftEffect::Neutral` on
   HP-41 hardware. Verify, then lock.

7. **`Op::Asto(u8)` vs `Op::Asto { reg: u8 }`** — existing ops use tuple
   variants (`Op::Sto(u8)`, `Op::Rcl(u8)`). Follow precedent.
