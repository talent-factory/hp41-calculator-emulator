# Tech-Debt — On-Screen-Keyboard kann bei offenem Label-Modal keine Buchstaben einspeisen

**Status:** RESOLVED-IN-v2.2.1 (2026-05-16)
**Resolution reference:** [`.planning/quick/260516-c1p-fix-gui-clp-binding-and-modal-letter-clicks/`](../quick/260516-c1p-fix-gui-clp-binding-and-modal-letter-clicks/)
**Resolution scope:** Option A (alphaChar-fallback in handleClick for xeq_name / clp / assign_label modals) plus the related CLP binding gap (mode-aware shifted √x in PRGM mode) plus doc patch to `docs/verifying-card-reader.md`.
**Discovered:** 2026-05-16, beim Eintippen des Quadratic-Solver-Tutorials (`PRGM → LBL → "QUAD"`) in der Tauri-GUI — und prompt eskaliert von "v3.x-Backlog" auf "v2.2.1-Hotfix", weil `docs/verifying-card-reader.md` (CLI ↔ GUI parity verification procedure) genau diese Tastenfolge fordert.
**Affected surface:** `hp41-gui` only — CLI (`hp41-cli`) ist nicht betroffen (Physische Tastatur funktioniert dort regulär).

## Symptom

Im PRGM-Modus öffnet `SHIFT + STO` korrekt das LBL-Prompt-Modal (LCD zeigt `LBL _`). Klicks auf On-Screen-Tasten zur Eingabe des Labelnamens haben jedoch keinen Effekt:

- **Klick auf `ALPHA`** → wird stillschweigend verworfen.
- **Klick auf Buchstabentasten** (z. B. `Σ-` mit `alphaChar: 'Q'`) → wird stillschweigend verworfen.
- **Workaround heute:** Buchstaben auf der physischen Tastatur tippen (`q u a d Enter`). Funktioniert via `hp41-gui/src/App.tsx:436-446` (1-Zeichen-`e.key` → `modalKey` → `handleModalKey`).

Betrifft drei Modal-Familien: `xeq_name` (LBL / XEQ / GTO), `assign_label` (ASN), und perspektivisch `clp` falls es On-Screen reachable wird.

## Root cause (verifiziert in v2.2)

`hp41-gui/src/App.tsx::handleClick` löst die `effectiveId` per Resolution-Regeln 2–4 auf:

```ts
if (alphaOn && key.alphaChar) {
  effectiveId = `alpha_${key.alphaChar}`;   // (a) ALPHA-Modus aktiv
} else if (shiftActive && key.shifted) {
  effectiveId = key.shifted.id;             // (b) shifted-Variante
} else {
  effectiveId = key.id;                     // (c) Primär-ID
}
```

Im offenen Label-Modal trifft Pfad (c) → `effectiveId` ist der Primary-Op-ID (`alpha_toggle`, `sigma_minus`, `xy_swap`, …). Dieser String wird in der Modal-Routing-Schicht ungefiltert an `handleModalKey` weitergegeben. Im `xeq_name`-Zweig (`pending_input.ts:355`) gilt:

```ts
if (isPrintableChar(key) && pending.acc.length < 24) { … }   // isPrintableChar: key.length === 1
```

Mehrzeichen-IDs wie `'alpha_toggle'` (Länge 12) oder `'sigma_minus'` (Länge 11) scheitern an dieser Predicate-Prüfung → Modal-State bleibt unverändert, kein User-Feedback, keine Toast-Meldung (D-07-Lückenfall).

Es gibt aktuell **keinen Pfad**, der eine On-Screen-Klick-ID in ein einzelnes Zeichen für das Modal übersetzt. Der `alphaChar`-Pfad in (a) springt nur, wenn der ALPHA-Annunciator schon aktiv ist — aber `alpha_toggle` selbst wird beim Klick im Modal verworfen, und das LBL-Modal ist nur ohne aktiven ALPHA-Annunciator erreichbar (D-25.5 / v2.1-Divergenz "ALPHA überschreibt SHIFT").

## Why this slipped through

- Phase 26 ("GUI Integration & Polish") hat die 13 Prompt-IDs frontendseitig auf React-Modale umgehängt und damit das Display korrekt befüllt (`LBL _` erscheint).
- E2E-Smoke (FN-QUAL-05, Phase 27) testet nur `2 ENTER 3 + → 5.0000` — kein Label-Modal-Pfad, kein On-Screen-Buchstabenklick.
- Vitest `pending_input.test.ts:376-384` deckt `lbl_X` ab, simuliert die Buchstabe aber direkt als `'X'`-String (so wie die physische Tastatur sie liefert), nicht als On-Screen-Klick-ID.
- Manuelles Dogfooding mit dem Quadratic-Tutorial wurde bisher offenbar über die physische Tastatur gemacht.

## Proposed fix (v3.x, NICHT Phase 28 sofort)

Drei Optionen, in Reihenfolge der Empfehlung:

**Option A — `alphaChar`-Fallback im Modal-Pfad (kleinster Diff, semantisch sauber):**
In `handleClick` (vor `routedKey`-Berechnung) ergänzen:

```ts
if (pendingInput !== null
    && (pendingInput.kind === 'xeq_name' || pendingInput.kind === 'assign_label')
    && key.alphaChar) {
  routedKey = key.alphaChar;
}
```

Vorteil: keine Veränderung am State-Modell. Nachteil: muss bei jedem neuen Modal-Kind, das Text akzeptiert, ergänzt werden. Empfohlen, ausreichend für v3.x-Scope.

**Option B — Implizites ALPHA-Toggling beim Modal-Open:**
LBL/XEQ/GTO setzen `alphaActive` (lokales Frontend-State, analog zu `shiftActive`) auf `true` für die Modal-Dauer; `handleClick`-Pfad (a) greift automatisch. Erfordert neuen Flag, aber konsistenter mit dem ALPHA→Buchstaben-Pattern.

**Option C — Hardware-treue ALPHA-Sequenz nachbauen:**
LBL öffnet kein Modal, sondern setzt einen `pending_label_op`-Flag. Erst ein anschließender Klick auf `ALPHA` aktiviert ALPHA-Modus, dann werden Buchstaben über den normalen `alpha_<X>`-Pfad eingegeben, und ein zweiter `ALPHA`-Klick beendet die Eingabe. Komplex (zwei Pfade durch das Modal-Modell), bricht die in Phase 26 etablierte Modal-LCD-Konvention. **Nicht empfohlen.**

## Acceptance criteria (falls in v3.x geplant)

- Klick auf Buchstabentasten im offenen LBL/XEQ/GTO-Modal hängt das jeweilige `alphaChar` an den Akkumulator an.
- `[A]`, `[B]`, …, `[J]` (mit explizitem `alphaChar`) UND die Buchstaben unterhalb der Bedien-Tasten (`Q`, `R`, `S`, `T`, `U`, …) sind alle erreichbar.
- `pending_input.test.ts` erhält eine neue Test-Suite, die On-Screen-Klick-IDs als Modal-Eingabe simuliert (nicht nur `'Q'`-Strings).
- Smoke-Test (`smoke.spec.ts`) wird um eine Label-Eingabe-Sequenz erweitert: `SHIFT → STO → click(Q) → click(U) → click(A) → click(D) → ENTER`, dann assert `program[0]` startet mit `LBL "QUAD"`.
