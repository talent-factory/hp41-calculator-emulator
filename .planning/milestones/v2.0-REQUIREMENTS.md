# Requirements Archive: HP-41 Calculator Emulator v2.0

**Archived:** 2026-05-10
**Milestone:** v2.0 Tauri GUI (Phases 13–18)
**Status:** ✅ All requirements completed (except PROG-01 partial: CI human verification deferred)

## v2.0 Requirements (All Shipped)

### WORKSPACE — Workspace & Infrastructure

- [x] **WSPC-01**: User can build and launch `hp41-gui` from the Cargo workspace via `just gui-dev`; `just ci` (CLI pipeline) still passes without modification
  - Phase 13 (all 3 plans)
  - Outcome: `hp41-gui` nested standalone workspace; `just gui-dev` opens HP-41 Calculator window; `just ci` remains green; root `Cargo.toml` members unchanged

- [x] **WSPC-02**: User can build both `hp41-cli` and `hp41-gui` in the same workspace without either binary's CI regressing
  - Phase 13 (Plans 13-01 + 13-03)
  - Outcome: Nested workspace isolation confirmed; `tauri`/`tauri-build` never enter root Cargo resolver; CI matrix unaffected

### SKIN — HP-41C Visual Skin

- [x] **SKIN-01**: User sees a pixel-perfect SVG HP-41C key layout (9×5 grid, ENTER double-width, correct key labels and legends, HP-41C proportions and color scheme — dark brown body, gold shift labels)
  - Phase 16 (Plans 16-01 + 16-02)
  - Outcome: `Keyboard.tsx` SVG with 44-key layout (9+8+9+9+9 rows), dark brown body, gold shift legends, white primary legends; viewBox scales on Retina/HiDPI

- [x] **SKIN-02**: User can click any key in the SVG skin and the corresponding HP-41 operation executes in `hp41-core` (same result as pressing the equivalent CLI key binding)
  - Phase 16 (Plans 16-01 + 16-02)
  - Outcome: `handleKeyClick` in `Keyboard.tsx` calls `dispatch_op` with correct key ID; all 23 named KEY_DEFS `id` values pass `key_map::resolve()` (validated by Wave 0 test)

- [x] **SKIN-03**: User sees visual press feedback (CSS scale-down animation) on every key click
  - Phase 16 (Plan 16-02)
  - Outcome: `pressedKey` state machine with `scale(0.92)` animation, 150ms `setTimeout`; `transform-box: fill-box` for SVG center-origin transform

### DISP — Display & Readouts

- [x] **DISP-01**: User sees the 12-char HP-41 display string and all five annunciators (USER, PRGM, ALPHA, RAD, GRAD) update in the GUI after every operation
  - Phase 15 (Plan 15-03)
  - Outcome: React `App.tsx` with `display_str` panel and 5 annunciator indicators; updates after every `dispatch_op` IPC call

- [x] **DISP-02**: User sees a stack register panel showing X/Y/Z/T/LASTX values alongside the calculator skin, updating after every operation
  - Phase 15 (Plans 15-02 + 15-03)
  - Outcome: `CalcStateView` extended with `y_str`/`z_str`/`t_str`/`lastx_str`; stack panel in `App.tsx` shows all 5 registers

### IPC — IPC & Core Integration

- [x] **IPC-01**: All user operations reach `hp41-core` via Tauri Rust commands (`dispatch_op`, `get_state`); the response is a `CalcStateView` (~200 bytes); `print_buffer` is drained on every command; no `hp41-core` logic is duplicated in the GUI crate
  - Phase 14 (all 4 plans)
  - Outcome: `dispatch_op` + `get_state` commands verified; `CalcStateView` ~170 bytes; `print_buffer` drained; SC-4 invariant (`grep -rn "fn op_\|fn flush_entry\|fn format_hpnum" hp41-gui/src-tauri/src/` returns nothing) confirmed

- [x] **IPC-02**: User can operate the GUI entirely from the physical keyboard using the same key bindings as `hp41-cli` (e.g. `0–9`, `+`, `-`, `*`, `/`, `ENTER`, `q`→SIN, etc.)
  - Phase 15 (Plans 15-02 + 15-03)
  - Outcome: `resolveKeyId` maps keyboard events to HP-41 op key IDs; `useCallback`+`useEffect` listener with `e.repeat` guard; all hp41-cli bindings covered

### PERS — Persistence

- [x] **PERS-01**: User's calculator state persists across GUI restarts via `~/.hp41/autosave.json` (shared with `hp41-cli`); state auto-saves every 30 seconds; save files created by v1.x `hp41-cli` load without error in `hp41-gui`
  - Phase 17 (Plans 17-01 + 17-03)
  - Outcome: `persistence.rs` in hp41-gui with `dirs` dep; 30s auto-save thread; startup load; v1.x CLI save files load without error (same JSON schema)

- [x] **PERS-02**: User sees PRX/PRA/PRSTK print output in a scrollable panel in the GUI (output from the `print_buffer` is surfaced, not silently discarded)
  - Phase 17 (Plans 17-02 + 17-03)
  - Outcome: Scrollable print panel with auto-show, history accumulation via `printLog` state, auto-scroll to bottom

### PROG — Program Display

- [x] **PROG-01**: User can view the current program listing and navigate steps with SST/BST in GUI PRGM mode
  - Phase 18 (all 4 plans)
  - Outcome: Conditional program listing panel (auto-show on `annunciators.prgm`); `format_all_steps()` produces step strings; `handle_sst`/`handle_bst` Tauri commands; F7/F8 keyboard bindings; auto-scroll to active step; cross-platform `ci-gui.yml` CI job (3-OS matrix, path filter, independent from `ci.yml`)
